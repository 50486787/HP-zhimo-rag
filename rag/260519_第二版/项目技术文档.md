```
│  │ Kimi k2.5 翻译                  │   │
│  │ bge-m3   embedding (Xinference) │   │

           │              │
           ▼              ▼
    ┌──────────┐   ┌─────────────┐
    │ RAGFlow  │   │ Xinference  │
    │ :9900    │   │ :9997       │
    │ 文档存储  │   │ bge-m3 模型  │
    └──────────┘   └─────────────┘
```

**核心思路：** RAGFlow 只做文档存储（不做检索），启动时一次性拉取全部文档的 content 列，用 bge-m3 全量 embedding 到内存矩阵。查询时矩阵点积打分，叠加文件名/标签的关键词权重，排序返回。

## 3. search_core.py — Searcher 核心

### 3.1 初始化

```python
searcher = Searcher(
    vocab_path,      # 规范词频 JSON 路径
    ragflow_key,     # RAGFlow API key
    ragflow_ds_ids,  # 数据集 ID 列表
    ragflow_base,    # http://127.0.0.1:9900/api/v1
    kimi_api_key,    # Kimi API key
    kimi_base_url,   # https://api.moonshot.cn/v1
    bge_base_url,    # http://127.0.0.1:9997/v1
)
searcher.initialize()  # 加载文档 + embedding ~5.8s
```

### 3.2 初始化流程 (`initialize()`)

1. `load_docs()` — 从 RAGFlow REST API 拉取全部数据集的所有 chunk
   - `GET /datasets/{id}/documents` 获取文档列表
   - `GET /datasets/{id}/documents/{doc_id}/chunks` 获取每个文档的 chunk
2. `_parse_chunk()` — 从 RAGFlow Table 模式的 chunk 中提取 path / filename / content 三列
3. `_embed()` — 对 `"{filename} {tags}"` 做 bge-m3 embedding（用 `requests.post` 直连，不用 OpenAI client，因 httpx 与 Xinference 代理不兼容）
4. `_normalize()` — L2 归一化，存为 `doc_matrix` (float32 numpy 矩阵)

### 3.3 搜索流程 (`search(query)`)

1. **Kimi 翻译** — 口语→结构化 JSON：`{style, keywords, vector_query}`
   - 受控词表约束关键词映射（如 "院子"→"庭院"）
   - 复合词拆分去元标签（如 "奶油风格"→"奶油"）
2. **Query embedding** — bge-m3 嵌入 `vector_query`，归一化
3. **矩阵点积** — `doc_matrix @ query_vec * 100` → 基准分
4. **分层加权** — 每个文档：
   - 文件名命中 keyword：+300/词（`_` 分词后逐段匹配）
   - 标签命中 keyword：+100/词
   - 最终分 = 基准分 + 文件名加权 + 标签加权
5. **分组排序** — keyword 命中组在前（按 final_score 降序），纯语义组在后（取 top 5）

### 3.4 关键方法

| 方法 | 说明 |
|------|------|
| `translate(query)` | Kimi 口语→结构化翻译 |
| `load_docs()` | 从 RAGFlow 拉全量 chunk |
| `initialize()` | 加载+embed+归一化，必须在使用前调用 |
| `search(query)` | 完整搜索，返回 `{style, keywords, vector_query, results, total, n_hit, n_semantic}` |
| `get_groups()` | 返回去重排序的分组列表（已去 P 前缀） |
| `get_group_items(group)` | 返回某分组下的全部索引（兼容 P 前缀） |

### 3.5 设计决策

- **不用 RAGFlow 检索** — RAGFlow 的检索权重不可控（混合检索比例、关键词匹配字段等全是黑盒）。一次性拉全量文档到内存，自己控制打分逻辑。
- **不用 rerank 模型** — 用户明确拒绝。单层 bge-m3 + 规则加权已满足需求，且无额外推理开销。
- **bge-m3 不用 OpenAI client** — Xinference 的 `/v1/embeddings` 端点对 httpx 大请求返回 502。直接用 `requests.post` + batch_size=200 稳定。
- **P 前缀** — RAGFlow 存 path 时加 "P" 前缀（"P01"），防止 Table 解析器把 "01" 当成数字。对外展示和文件系统访问时去掉 P 前缀。

## 4. step06_web_server.py — FastAPI 服务

### 4.1 路由表

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/` | 返回 `static/index.html` |
| GET | `/api/groups` | 返回所有分组列表 `{groups: [...]}` |
| GET | `/api/browse?group=&page=&page_size=` | 浏览某分组下的文档（分页） |
| GET | `/api/search?q=` | AI 搜索 |
| GET | `/api/detail?path=&filename=` | 单条文档详情（含 jpg/zip 是否存在） |
| GET | `/img/{rest_path}?w=` | 图片代理：NAS 原图→缩略图→缓存 |
| GET | `/download/{rest_path}` | ZIP 文件下载 |

### 4.2 安全设计

- **路径遍历防护** — `_safe_path()` 用 `os.path.realpath` + `os.path.commonpath` 校验，禁止 `../` 逃逸
- **P 前缀映射** — `_real_path()` 去掉 P 前缀后拼 NAS 路径
- **503 保护** — `_ensure_ready()` 在 Searcher 未初始化时返回 503
- **XSS 防护** — 前端 `esc()` 转义全部用户可控内容，`encodeURIComponent` 编码 URL 参数

### 4.3 缩略图缓存

- 目录：`.thumb_cache/`
- 缓存键：`{safe_name}_{width}.jpg`
- 原子写入：先写 `.tmp` 文件再 `os.replace()`
- 源：Pillow `Image.LANCZOS` 等比缩放，JPEG quality=80
- 无过期清理（一期不处理，图片量不大）

### 4.4 启动

```bash
# 设置 NAS 路径（环境变量或交互输入）
$env:NAS_BASE = "H:\claude_code\rag\260519_第二版\知末精标"
```

## 5. static/index.html — 前端

### 5.1 技术选型

- **零依赖** — 纯 HTML/CSS/JS，无框架，无构建步骤
- **深色主题** — 背景 `#1a1a2e`，卡片 `#16213e`，强调色 `#e94560`
- **CSS Grid 瀑布流** — 5/4/3/2 列响应式断点，左→右行优先排列

### 5.2 状态管理

```javascript
state = {
  groups: [],          // 分组列表
  activeGroup: null,   // 当前选中分组
  mode: 'browse',      // 'browse' | 'search'
  searchResults: null, // 搜索结果缓存
  requestId: 0,        // 递增计数器，防止竞态
}
```

### 5.3 交互流程

1. **页面加载** → `GET /api/groups` → 渲染分组标签 → 自动选中最新分组 → `GET /api/browse` → 瀑布流展示
2. **切换分组** → 清空搜索 → `GET /api/browse?group=X` → 重新渲染
3. **搜索** → `GET /api/search?q=X` → 渲染搜索结果 + keyword 标签 + 图例说明
4. **点击卡片** → `GET /api/detail?path=&filename=` → 弹窗显示大图 + 标签 + 下载按钮
5. **下载** — 卡片和弹窗都有下载按钮，卡片上 `stopPropagation` 防止触发展开

### 5.4 视觉标记

- 文件名命中 keyword → 卡片边框/文字变红（`.hit-fn`）
- 标签命中 keyword → 卡片上风格标签变橙（`.hit-tag`）
- 搜索信息栏显示解析出的关键词和风格

## 7. 架构决策记录

### 为什么不用 RAGFlow 检索

RAGFlow 的检索接口返回的是混合检索结果，权重配置（关键词 vs 向量、chunk 粒度、相似度阈值）全部黑盒。景观设计搜索需要精确控制"文件名匹配 > 标签匹配 > 语义相似"的优先级，只有自己全量加载+自定义打分才能做到。

### 为什么 bge-m3 不用 OpenAI client

`openai.OpenAI(base_url="http://127.0.0.1:9997/v1").embeddings.create()` 底层用 httpx。当请求体较大（1300+ 条文本），httpx 触发 Xinference 的 502 Bad Gateway（响应体为空）。直接 `requests.post` + 分批次 200 条稳定。

### 为什么是 P 前缀

RAGFlow 的 Table 解析模式会在文本开头加 `- path:` 前缀。如果 path 值是 "01"，解析器可能把 "01" 当成数字序号而非字符串。加 "P" 前缀（"P01"）强制当作字符串。对外展示和文件系统访问时去掉 P。

### 为什么文件内聚在 260519_第二版/

保持 v2 产出的工作区隔离。原 step01~04 文件和 v1 搜索逻辑作为只读备份留在同级目录，不修改也不引用。
