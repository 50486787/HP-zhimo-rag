# v2 搜图架构设计 — BGE 单层检索 + 分层加权

## 背景

v1 搜图链路: Kimi翻译 → RAGFlow向量检索 → LLM裁判重排。受知乎文章《从Embedding困局到确定性返回》启发，v2 改造核心思路：

- **不用 rerank**（用户反馈精度差）
- **RAGFlow 降级为纯数据存储**，不做检索
- **bge-m3 embedding 做基础分**，配合确定性规则加权
- **文件名 = 标题层**，权重最高（人工精选的关键词组合）

## 架构总览

```
启动阶段:
  RAGFlow chunks列表API → 全量1360条文档 → bge-m3 批量embed → 存储内存(1360×1024)

查询阶段:
  用户口语
     ↓
  Kimi k2.5 翻译 → {style, keywords[], vector_query}
     ↓
  bge-m3 embed(vector_query) → 1×1024
     ↓
  与预计算的 N×1024 矩阵做余弦相似度 → ×100 → 基础分 [0~100]
     ↓
  确定性加权:
    - 文件名命中 keyword → +300/词（叠加）
    - 标签命中 keyword → +100/词（叠加）
    - 仅对长度 ≥2 的关键词生效
     ↓
  排序 → Top-K → 拼接NAS路径 → 展示 jpg + zip
```

## 组件职责

| 组件 | 角色 | 接口 |
|------|------|------|
| **RAGFlow** | 增量数据库 + 管理后台 | chunks列表API（全量拉取） / retrieval API（不入检索） |
| **Xinference bge-m3** | 向量编码器 | /v1/embeddings |
| **Kimi k2.5** | 口语翻译 + 关键词提取 | OpenAI-compatible API |
| **search_v2.py** | 编排上述三个服务 | — |

## 数据流

### 启动加载

1. 调 RAGFlow dataset documents API 列出所有文档
2. 对每个文档调 chunks 列表 API 拉全量（不需要 question，不走检索过滤）
3. 解析每条 chunk: path, filename, tags
4. 构造 doc_text = filename + " " + tags
5. 调 bge-m3 批量 embed → N×1024 矩阵存内存
6. 打印 "已加载 N 条文档"

### 查询

1. Kimi 翻译口语 → `{style, keywords[], vector_query}`
2. bge-m3 embed vector_query → 1×1024
3. numpy 矩阵点积 → cosine × 100 → 基础分
4. 遍历每条文档:
   - 文件名命中任意 keyword → +300（叠加）
   - 标签命中任意 keyword → +100（叠加）
5. argsort 降序 → Top-K (默认10)
6. path 拼接 NAS base → 展示 jpg + zip

## 打分公式

```
最终分 = cosine(query_vec, doc_vec) × 100
       + Σ(filename命中keyword ? 300 : 0)
       + Σ(tags命中keyword ? 100 : 0)
```

- cosine 值域 [-1, 1] → ×100 映射到 [-100, 100]
- 文件名每命中一个 keyword +300，多关键词命中叠加
- 标签每命中一个 keyword +100，多关键词命中叠加
- 匹配仅对长度 ≥2 的关键词生效，避免单字误匹配
- 文件名按 `_` 分词后逐段匹配，每 keyword 每文件最多命中一次
- 已知局限：2字短词可能误匹配为长词的一部分（如"客厅"误中"会客厅"），无词典无法根治，当前影响可控

## Kimi 翻译 Prompt 要点

- 风格未提则留空，**不要猜测**
- keywords **必须保留用户原话的实体词**，不能丢失或替换
- vector_query 用 keywords 拼接，**不加无关维度**
- **复合词必须拆分**：如"奶油风格"→"奶油"+"风格"，避免绑死导致文件名不命中

## 关键决策

1. **不用 bge-reranker-v2-m3** — cross-encoder 精度差且每次 1360 次 forward 太慢
2. **不用 RAGFlow 做检索** — RAGFlow 退化为纯数据存储，提供增量管理和 Web 后台
3. **分层打分来自文章启发** — 标题(文件名) = 最重信号，标签 = 中等信号，embedding = 兜底
4. **文件名命中用 Python 字符串匹配判定** — 确定性、可审计，不受 embedding 不确定性影响
5. **启动用 chunks 列表 API 不是检索 API** — 检索 API 需要 question 且会过滤低相似度文档，chunks API 能拿全量
